## 二、角色详解

### 2.1 人员 A —— 前端工程师（Frontend Engineer）

**一句话定位**：负责玩家看到的所有东西，以及玩家与游戏交互的每一个环节。

#### 核心职责

| 优先级 | 职责 | 说明 |
|---|---|---|
| P0 | 游戏主界面实现 | 文本展示区、选择面板、玩家状态面板的布局与交互 |
| P0 | 打字机效果 | 叙事文本逐字显示，支持速度调节、跳过 |
| P0 | 前后端联调 | 调用后端 API，处理响应，更新界面状态 |
| P1 | 动效系统 | 场景切换动画、数值变化飘字、按钮反馈 |
| P1 | 存档/读档界面 | 存档列表、读档确认、自动存档提示 |
| P2 | 调试面板 | 开发期专用的游戏状态查看器、强制触发事件 |
| P2 | 响应式适配 | 确保在不同屏幕尺寸下基础可用 |

#### 技术栈

- React 18 + TypeScript
- Tailwind CSS（样式）
- Zustand（状态管理）
- Axios / Fetch（API 调用）

#### 交付物清单

| 交付物 | 截止时间 | 验收标准 |
|---|---|---|
| 前端项目骨架 | Day 1 | `npm run dev` 可启动，目录结构清晰 |
| 类型定义文件 | Day 2 | 与 GameState Schema 100% 对齐 |
| API Client 封装 | Day 3 | 封装所有后端接口，含错误处理 |
| 核心 UI 组件 | Day 4 | TextDisplay、ChoicePanel、StatusPanel 可用 |
| 完整交互流程 | Day 5 | 能完成一次开始游戏 -> 选择 -> 状态变化 |
| 存档/读档 UI | Day 12 | 能完整操作存档和读档 |
| 动效系统 | Day 9 | 打字机、数值飘字、场景切换 |

#### 需要依赖的输入

- **Day 2 从人员 B 获取**：`GameState Schema` → 用于定义前端类型
- **Day 3 从人员 B 获取**：`API 契约` → 用于实现 API Client
- **Day 5 从人员 B 获取**：可运行的后端服务 → 用于联调

#### 给人员 A 的特别提醒

> 文字游戏的前端工作量**一点都不小**。玩家的沉浸感 70% 来自文字呈现的节奏和界面反馈。不要觉得"只是展示文字"，打字机的速度、选择出现的时机、状态变化的可视化，这些细节决定游戏质感。

### 2.2 人员 B —— 后端工程师 + 游戏系统设计师（Backend & Game Engine）

**一句话定位**：项目的"骨架搭建者"，定义所有数据的结构和流转规则，实现游戏核心逻辑。

#### 核心职责

| 优先级 | 职责 | 说明 |
|---|---|---|
| P0 | 接口契约定义 | 定义 GameState Schema 和 API 规范，全组共享 |
| P0 | API 服务 | FastAPI 搭建，Session 管理，路由实现 |
| P0 | 游戏状态管理 | GameState 的创建、更新、序列化、校验 |
| P0 | 数值系统 | 修为计算、境界突破判定、属性变化 |
| P0 | 事件系统 | 事件触发条件评估、分支处理、状态变更应用 |
| P1 | 存档系统 | 手动存档、自动存档、读档、存档列表 |
| P1 | 配置热加载 | 修改内容配置（YAML）无需重启服务 |
| P2 | NPC 关系系统 | 好感度变化、社交图谱基础 |
| P2 | 部署配置 | Docker Compose、环境变量、生产环境准备 |

#### 技术栈

- Python 3.11+
- FastAPI（Web 框架）
- SQLAlchemy（ORM）
- PostgreSQL（数据库）
- Pydantic（数据校验）
- PyYAML（配置解析）

#### 交付物清单

| 交付物 | 截止时间 | 验收标准 |
|---|---|---|
| 后端项目骨架 | Day 1 | `uvicorn main:app --reload` 可启动 |
| GameState Schema | Day 2 | 文档完整，字段含类型/范围/默认值，全组确认 |
| API 契约文档 | Day 3 | 所有端点含 Request/Response/错误码示例 |
| 数据库模型 | Day 2 | SQLAlchemy 模型与 Schema 一致 |
| 游戏引擎核心 | Day 4 | 数值计算、事件触发、状态更新可独立运行 |
| 端到端流程 | Day 5 | /session/start -> /action -> 返回新状态 可跑通 |
| 存档系统 | Day 12 | 存/读 10 次无错误，跨版本兼容 |
| 配置热加载 | Day 9 | 修改 YAML 后 5 秒内生效 |

#### 关键产出：GameState Schema

人员 B 最重要的产出是 **GameState Schema**。这是全组的"唯一事实来源"：
- 前端靠它渲染玩家状态
- Agent 靠它生成叙事
- 存档靠它序列化数据

**Schema 必须在 Day 2 锁定**，之后变更成本极高。

#### 需要依赖的输入

- **Day 2 从人员 D 获取**：内容配置 Schema 初稿 → 确保 GameState 能容纳内容数据
- **Day 4 从人员 C 获取**：Agent IO 格式 → 设计 Agent 调用接口
- **Day 9 从人员 D 获取**：完整内容配置 → 接入引擎测试

#### 给人员 B 的特别提醒

> 你是项目的**关键路径**。你的 GameState 和 API 没定下来，前端和 Agent 都没法全力开工。Day 1-Day 3 你的优先级高于一切。同时，你不仅要写代码，还要做"游戏系统设计"——修为怎么涨、境界怎么突破、事件怎么触发，这些规则需要你来设计。

### 2.3 人员 C —— Agent 工程师（Narrative Agent Engineer）

**一句话定位**：游戏的"叙事大脑"，让 LLM 根据游戏状态和世界设定生成有沉浸感的修仙故事。

#### 核心职责

| 优先级 | 职责 | 说明 |
|---|---|---|
| P0 | LLM 适配层 | 统一接口调用 OpenAI / Claude / 本地模型 |
| P0 | Prompt 工程 | 设计动态 Prompt 模板，注入 GameState + 角色卡 + 世界观 |
| P0 | 输出解析器 | 将 LLM 的文本输出解析为结构化 JSON |
| P0 | 记忆管理 | 维护对话历史，确保 NPC "记得"之前发生的事 |
| P1 | 叙事控制器 | 风格一致性检查、OOC（脱离设定）检测 |
| P1 | 降级机制 | LLM 失败/超时时返回预设文案，保证游戏不卡死 |
| P2 | 世界知识注入 | 将相关世界观知识动态注入 Prompt |
| P2 | 内容安全过滤 | 敏感词过滤、输出审核 |

#### 技术栈

- Python 3.11+
- LiteLLM（多模型统一接口）
- Jinja2（Prompt 模板）
- 复用团队已有 Agent 框架（LLM 适配层、基础工具调用）

#### 交付物清单

| 交付物 | 截止时间 | 验收标准 |
|---|---|---|
| Agent 项目骨架 | Day 1 | 目录结构清晰，能独立运行 |
| LLM Adapter | Day 3 | 能调用 LLM 并返回文本 |
| Agent IO 格式文档 | Day 4 | 输入输出 JSON Schema 定义完整 |
| Prompt Builder | Day 4 | 能动态注入 GameState 和角色卡 |
| 输出 Parser | Day 4 | 解析成功率 > 90%（20 次测试） |
| 完整 Agent 流程 | Day 5 | 接收输入 -> 生成叙事 -> 输出结构化 JSON |
| 记忆管理 | Day 8 | 对话历史正确维护，NPC 不"失忆" |
| 降级机制 | Day 12 | LLM 失败时 3 秒内返回预设文案 |

#### Agent 输入输出示例

**输入**（后端传给 Agent）：
```json
{
  "system_prompt": "你是一位修仙小说作者...",
  "context": {
    "scene": { "id": "trial_grounds", "description": "试炼场..." },
    "game_state": { /* 当前 GameState */ },
    "player_input": "将手放在测灵石上",
    "memory": {
      "dialogue_history": ["你来到试炼场", "测灵石发出微光"]
    }
  }
}
```

**输出**（Agent 返回给后端）：
```json
{
  "narrative": "你将手放在测灵石上，一股温热的气息顺着手臂流入体内...",
  "state_changes": {
    "player.cultivation": 10,
    "world.flags.trial_completed": true
  },
  "next_scene_id": "trial_result",
  "available_choices": [
    { "id": "express_gratitude", "text": "向长老道谢" },
    { "id": "stay_silent", "text": "默默退下" }
  ]
}
```

#### 需要依赖的输入

- **Day 2 从人员 B 获取**：GameState Schema → 知道 Prompt 里能注入哪些状态
- **Day 2 从人员 D 获取**：世界观文档 → 了解修仙体系，写入 system_prompt
- **Day 3 从人员 D 获取**：NPC 角色卡 → 注入角色性格和对白风格
- **Day 4 从人员 B 获取**：后端 Agent 调用接口 → 确定如何被后端调用

#### 给人员 C 的特别提醒

> 你不是"调 API 的"，你是**叙事设计师**。你必须深入理解修仙世界观，知道什么时候该写"灵气运转"、什么时候该写"突破瓶颈"。Prompt 设计的好坏直接决定游戏质感。同时，LLM 不稳定是常态，你的 Parser 必须足够健壮，做好容错和降级。

### 2.4 人员 D —— 世界观设计师 + 内容工程师（Worldbuilder & Content Engineer）

**一句话定位**：修仙世界的"造物主"，设计一切设定，并用配置文件将内容注入游戏。

#### 核心职责

| 优先级 | 职责 | 说明 |
|---|---|---|
| P0 | 世界观设定 | 修仙体系、门派历史、地理、规则 |
| P0 | 内容配置 Schema | 定义 NPC、事件、物品的 YAML 结构规范 |
| P0 | NPC 角色卡 | 为每个 NPC 编写性格、背景、对话风格配置 |
| P0 | 事件剧本 | 编写事件流程、分支选择、触发条件 |
| P1 | 物品/功法配置 | 定义物品属性、效果、获取途径 |
| P1 | 配置校验工具 | 编写脚本自动检查配置格式和逻辑错误 |
| P2 | 内容编辑器 | 开发简易工具（Web 或 Python）可视化编辑配置 |
| P2 | 游戏文案 | 所有玩家-facing 的文本（提示、错误信息、UI 文案） |

#### 技术栈

- YAML / JSON（配置文件）
- JSON Schema（结构校验）
- Python（校验脚本、内容工具）
- Markdown（世界观文档）

#### 交付物清单

| 交付物 | 截止时间 | 验收标准 |
|---|---|---|
| 世界观文档 | Day 2 | 修仙体系、门派设定、历史背景完整 |
| 内容配置 Schema | Day 2 | JSON Schema 文件，能通过校验工具 |
| NPC 角色卡（3个） | Day 3 | 通过 validate_content.py，含性格/背景/对话风格 |
| 物品配置 | Day 4 | 通过校验，含属性/效果 |
| 事件配置（5-8个） | Day 10 | 通过校验，含完整分支和触发条件 |
| 校验脚本 | Day 2 | 能检测 YAML 语法错误、必填字段缺失 |
| 内容编辑器 | Day 8 | 能可视化编辑事件并导出 YAML |
| 游戏内文案 | Day 12 | 所有玩家可见文本完整、无错别字 |


## 五、常见问题解答

### Q1：如果某人进度落后怎么办？

**答**：
1. 站会上坦诚说明，全组一起评估影响
2. 优先保障关键路径（人员 B 的接口契约不能拖）
3. 非关键任务可以延后或砍掉
4. 必要时调整范围（如减少事件数量），但不能拖过 Day 5 里程碑

### Q2：接口契约定下来后发现有问题怎么办？

**答**：
1. 小问题：在群里说明，双方协商后文档和代码一起改
2. 大问题：开 15 分钟紧急会议，评估影响范围，决定是否变更
3. **原则**：契约文档优先于代码。文档改了，代码必须跟着改。

### Q3：Agent 生成的叙事质量很差怎么办？

**答**：
1. 人员 C 优化 Prompt（加入更多示例、约束）
2. 人员 D 检查角色卡是否足够详细（性格、对话风格示例）
3. 如果 LLM 本身能力不足，准备降级方案（预设文案 + 少量动态填充）
4. 人员 B 确保注入的 GameState 信息完整准确

### Q4：内容配置和游戏引擎对不上怎么办？

**答**：
1. 人员 D 的校验脚本应该在提交前捕获大部分问题
2. 运行时错误由人员 B 和人员 D 一起排查：是配置格式问题还是引擎解析问题？
3. 如果是 Schema 理解不一致，立即更新文档并同步全组

### Q5：前端和后端数据不一致怎么办？

**答**：
1. 打开浏览器的 Network 面板，对比后端返回的 JSON 和前端显示的数据
2. 检查前端类型定义是否与 GameState Schema 一致
3. 如果是 API 返回错误，人员 B 排查；如果是前端解析错误，人员 A 排查
4. **黄金法则**：以数据库/后端的数据为准，前端只是"展示层"
