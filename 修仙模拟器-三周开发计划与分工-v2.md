> **版本**: v2.0

## 1. 项目概述与 MVP 范围

### 1.1 项目目标
在三周内交付一个**可运行的文字修仙模拟器 MVP**。核心体验：玩家扮演初入修仙门派的弟子，通过**预设选择 + 自由输入**推动剧情，Agent 基于玩家状态、NPC 角色卡和世界设定**实时生成独一无二的叙事文本**。

**关键设计原则**：
- **事件系统** = 编剧：决定"发生什么"（修为+10、好感度-5、进入新场景）
- **Agent** = 作家：决定"怎么呈现"（场景描写、NPC 对话、玩家心理活动）
- **同一事件，每次体验都不同**：LLM 根据 GameState 动态生成叙事，而非背诵预设文案

### 1.2 MVP 范围（严格守住，不能膨胀）

| 模块 | MVP 包含 | MVP 明确不包含 |
|---|---|---|
| **场景** | 1 条主线：入门试炼 -> 拜师 -> 首次修炼 -> 门派小比 | 多分支主线、随机事件、开放世界 |
| **NPC** | 3 个角色：师父（玄清真人）、大师兄、同门对手 | 全门派生态、NPC 自主行动 |
| **数值系统** | 修为值、境界（练气1-3层）、灵根属性 | 战斗系统、功法树、炼丹炼器 |
| **事件系统** | 5-8 个事件节点，含分支选择 | procedurally generated 事件链 |
| **Agent 叙事** | 场景叙事生成、NPC 动态对话、玩家自由输入理解 | 世界书动态检索、长期记忆摘要、多 Agent 协作 |
| **自由输入** | 玩家可输入任意文本，Agent 理解意图并回应 | 复杂意图推理（如"暗示师兄帮我作弊"） |
| **前端** | 打字机效果、状态面板、选择分支、自由输入框、存档/读档 | 动画特效、主题皮肤、移动端适配 |
| **存档** | 单存档槽、手动存档+自动存档 | 多存档管理、云存档 |

**铁律**：三周内的任何新想法，先问"这是不是 MVP 必须的？"，如果不是，记入 `docs/backlog.md`。

### 1.3 可复用资产
团队此前已构建一套通用 AI Agent 框架（含 LLM 适配层、RAG Pipeline、Tool Calling、Plugin 机制、99 个通过测试）。**Agent 层的核心基础设施可直接复用**，人员C在此基础上专注 Prompt 工程与叙事控制逻辑，而非从零搭建框架。

### 1.4 "固定事件 + 开放叙事"混合模式详解

这是三周 MVP 最核心的架构决策，必须在全组达成共识：

```
┌─────────────────────────────────────────────────────────────────────┐
│                        事件系统（固定骨架）                           │
│  事件: 入门试炼                                                       │
│  触发条件: scene_id == "trial_grounds"                               │
│  逻辑结果:                                                           │
│    - player.cultivation += 10                                        │
│    - world.flags.trial_completed = true                              │
│  可用选择: [A]道谢 [B]退下                                            │
│  NPC 反应规则: master.affinity += 3 (选A) / -= 5 (选B)              │
└─────────────────────────────────────────────────────────────────────┘
                                  │
                                  │ 提供: GameState + 事件上下文 + 角色卡
                                  ▼
┌─────────────────────────────────────────────────────────────────────┐
│                        Agent 叙事层（完全开放）                        │
│  LLM 根据注入的信息动态生成：                                          │
│    - 场景描写（测灵石的颜色、光芒、温度）                              │
│    - NPC 对话（基于角色卡 + 当前好感度）                              │
│    - 玩家心理活动（基于历史选择）                                     │
│    - 自由输入回应（理解玩家意图并映射到游戏动作）                       │
└─────────────────────────────────────────────────────────────────────┘
```

**为什么这样设计**：
1. **引擎保障逻辑正确**：不会出现"练气期御剑飞行"的幻觉
2. **Agent 保障沉浸体验**：同一事件每次叙事都不同，NPC 对话完全个性化
3. **三周可交付**：固定骨架降低测试复杂度，开放叙事提供 Agent 价值

## 2. 团队组织架构与核心职责

### 2.1 角色定义

| 角色 | 负责人 | 核心职责 | 次要职责 |
|---|---|---|---|
| **前端工程师** | 人员A | Web 前端架构、交互体验、前后端联调 | UI 风格定义 |
| **后端工程师** | 人员B | API 服务、数据库、游戏引擎、**接口契约定义** | 部署脚本 |
| **Agent 工程师** | 人员C | LLM 叙事引擎、Prompt 体系、记忆管理、**自由输入处理** | Agent 调试工具 |
| **世界观设计师** | 人员D | 世界观设定、内容配置、NPC 角色卡、**内容工具** | 测试内容自测 |

### 2.2 关键协作原则

1. **契约先行**：任何跨模块接口必须先写成文档，双方 Review 通过后才能编码
2. **Mock 先行**：依赖方未完工时，先提供 Mock 数据并行开发
3. **每日同步**：15 分钟站会，只同步三件事：昨天完成、今天计划、是否阻塞
4. **阻塞升级**：被阻塞超过 4 小时，必须 @ 相关责任人 + 抄送全组
5. **分支隔离**：每人独立 Git 分支，通过 PR 合并，至少 1 人 Review

### 3.2 核心交互流程

#### 3.2.1 预设选择流程

```
玩家点击选项 [A]
    │
    ▼
前端 POST /api/session/{id}/action
    { action_type: "choice", payload: "choice_A_id" }
    │
    ▼
后端 ──► 事件系统：查找 choice_A 对应的 effects
    │     应用状态变更（修为+10、flag 设置）
    │
    ▼
后端 ──► Agent 桥接：组装 Prompt（新 State + 角色卡 + 上下文）
    │
    ▼
Agent ──► LLM 生成叙事文本 + 可用选项
    │
    ▼
后端 ◄── 解析 Agent 输出，提取 narrative + available_choices
    │
    ▼
后端返回 Response
    { new_state, narrative, available_choices }
    │
    ▼
前端 ──► 打字机显示 narrative
     ──► 更新状态面板
     ──► 渲染新的 available_choices
```

#### 3.2.2 自由输入流程（关键 Agent 体验）

```
玩家在输入框输入："长老，弟子有一事不明"
    │
    ▼
前端 POST /api/session/{id}/action
    { action_type: "free_input", payload: "长老，弟子有一事不明" }
    │
    ▼
后端 ──► 引擎：确认当前场景允许自由输入
    │
    ▼
后端 ──► Agent 自由输入处理器：
    │     Step 1: 理解意图（classify_intent）
    │             → "询问" / "请求" / "闲聊" / "挑衅" / "无关"
    │     Step 2: 映射到游戏动作（map_to_action）
    │             → 如果是"询问修炼方法"：master.affinity +2
    │             → 如果是"挑衅"：master.affinity -10
    │     Step 3: 生成叙事回应
    │
    ▼
Agent ──► LLM 生成个性化回应（基于角色卡 + 当前好感度）
    │
    ▼
后端 ◄── 解析输出，应用状态变更
    │
    ▼
后端返回 Response（与预设选择相同的格式）
    │
    ▼
前端 ──► 显示叙事文本 + 更新后的选项
```

## 4. 核心接口契约（必须严格执行）

### 4.1 GameState Schema（全局唯一事实来源）

由人员B在 **Day 2** 定义完成，全组 Review 通过后锁定。

```typescript
// GameState - 后端定义，前后端+Agent 共用
interface GameState {
  // 基础信息
  session_id: string;
  current_scene_id: string;
  turn_count: number;

  // 玩家属性
  player: {
    name: string;
    cultivation: number;        // 修为值 (0-10000)
    realm: {
      major: "练气" | "筑基";    // 大境界（MVP 仅练气）
      minor: number;            // 层数 (1-9)
    };
    spirit_root: {
      type: "金" | "木" | "水" | "火" | "土" | "杂灵根";
      quality: number;          // 品质 1-10
    };
    attributes: {
      strength: number;
      agility: number;
      intelligence: number;
      perception: number;
    };
    inventory: InventoryItem[];
  };

  // NPC 状态
  npcs: Record<string, NPCState>;

  // 世界状态
  world: {
    current_location: string;
    time: {
      day: number;
      period: "早晨" | "上午" | "下午" | "傍晚" | "夜晚";
    };
    flags: Record<string, boolean>;  // 剧情标志位
  };

  // 最近事件历史（用于 Agent 记忆，保留最近 10 条）
  recent_events: EventRecord[];

  // 自由输入历史（新增，用于 Agent 理解玩家风格）
  free_input_history: FreeInputRecord[];
}

interface NPCState {
  id: string;
  name: string;
  affinity: number;             // 好感度 (-100 ~ 100)
  location: string;
  known_info: string[];         // 玩家已知的关于该 NPC 的信息
  dialogue_history: string[];   // 与该 NPC 的最近对话记录（保留最近 5 条）
}

interface EventRecord {
  turn: number;
  scene_id: string;
  narrative: string;
  player_choice: string;
  state_changes: Partial<GameState>;
  timestamp: string;
}

interface FreeInputRecord {
  turn: number;
  input_text: string;
  interpreted_intent: string;   // Agent 理解的意图
  narrative_response: string;   // Agent 的回应
  timestamp: string;
}

interface InventoryItem {
  item_id: string;
  name: string;
  quantity: number;
  effects: ItemEffect[];
}
```

#### 4.5.3 自由输入处理流程

```yaml
1. 后端接收 free_input
   │
   ▼
2. 引擎检查：当前场景是否允许自由输入
   │   - 如果不允许：返回错误 + 提示"当前只能做出选择"
   │
   ▼
3. 后端调用 Agent：classify_intent
   │   Input: 玩家输入文本 + 当前场景 + 对话历史
   │   Output: { intent: "ASK", target: "master", topic: "cultivation", confidence: 0.92 }
   │
   ▼
4. 后端根据意图查询映射规则
   │   - 找到匹配的规则 -> 确定状态变更（如 affinity +2）
   │   - 找不到规则 -> 走默认处理（affinity 不变，通用回应）
   │
   ▼
5. 后端调用 Agent：generate_response
   │   Input: 意图 + 映射结果 + 角色卡 + GameState
   │   Output: 叙事文本 + NPC 回应
   │
   ▼
6. 后端应用状态变更，返回前端
```

### 5.2 记忆管理策略

#### 5.2.1 短期记忆（对话历史窗口）

```python
# agent/src/memory.py

class ShortTermMemory:
    """短期记忆：保留最近 N 轮交互，直接注入 Prompt"""
    
    def __init__(self, max_turns: int = 5):
        self.max_turns = max_turns
        self.history = []
    
    def add(self, turn: int, player_input: str, narrative: str):
        self.history.append({
            "turn": turn,
            "player_input": player_input,
            "narrative": narrative[:200]  # 截断，避免 Prompt 过长
        })
        if len(self.history) > self.max_turns:
            self.history.pop(0)
    
    def get_context(self) -> List[Dict]:
        return self.history
```

```python
class NPCMemory:
    """NPC 个体记忆：每个 NPC 维护自己的对话历史"""
```

### 5.3 输出解析与容错

```python
# agent/src/parser.py

class AgentOutputParser:
    """解析 Agent 输出，处理各种格式错误"""
    
    def parse(self, raw_output: str, max_retries: int = 3) -> Dict:
        """
        解析流程：
        1. 尝试直接 JSON 解析
        2. 如果失败，尝试从 Markdown 代码块中提取 JSON
        3. 如果失败，尝试用正则提取关键字段
        4. 如果全部失败，返回降级响应
        """
        for attempt in range(max_retries):
            try:
                # 尝试 1: 直接解析
                return json.loads(raw_output)
            except json.JSONDecodeError:
                pass
            
            try:
                # 尝试 2: 提取 Markdown 代码块
                json_match = re.search(r'```json\s*(.*?)\s*```', raw_output, re.DOTALL)
                if json_match:
                    return json.loads(json_match.group(1))
            except json.JSONDecodeError:
                pass
            
            try:
                # 尝试 3: 提取花括号包裹的内容
                json_match = re.search(r'\{.*\}', raw_output, re.DOTALL)
                if json_match:
                    return json.loads(json_match.group(0))
            except json.JSONDecodeError:
                pass
        
        # 全部失败：返回降级响应
        return self._fallback_response(raw_output)
    
    def _fallback_response(self, raw_output: str) -> Dict:
        """降级响应：使用原始文本作为 narrative，保持游戏可运行"""
        return {
            "narrative": raw_output[:500] if raw_output else "（叙事生成中...）",
            "narrative_segments": [
                {"type": "narration", "text": raw_output[:500] if raw_output else "..."}
            ],
            "state_changes": {},
            "available_choices": [],
            "parse_failed": True
        }
```

### 5.4 降级机制

```python
# backend/src/services/agent_bridge.py

class AgentBridge:
    """Agent 桥接服务：处理 LLM 调用失败时的降级"""
    
    async def generate_narrative(self, context: Dict) -> Dict:
        try:
            # 正常调用 Agent
            response = await self.agent.generate(context, timeout=5.0)
            parsed = self.parser.parse(response)
            
            # 校验 parsed 输出
            if self._validate_output(parsed):
                return parsed
            else:
                raise ValueError("Output validation failed")
                
        except (TimeoutError, ValueError, Exception) as e:
            # 降级：使用预设文案 + 动态填充
            return self._degraded_response(context, str(e))
    
    def _degraded_response(self, context: Dict, error: str) -> Dict:
        """
        降级响应策略：
        1. 查找当前场景的预设文案
        2. 用 GameState 中的数据做简单替换
        3. 保证游戏可以继续进行
        """
        scene_id = context["current_scene"]["id"]
        preset = self.preset_loader.get(scene_id, "default")
        
        # 简单模板替换
        narrative = preset["narrative"].format(
            player_name=context["game_state"]["player"]["name"],
            cultivation=context["game_state"]["player"]["cultivation"]
        )
        
        return {
            "narrative": narrative,
            "state_changes": context.get("event_context", {}).get("triggered_effects", {}),
            "available_choices": preset.get("choices", []),
            "degraded": True,
            "error": error
        }
```

### 7.3 Mock 使用规则

1. **Mock 必须符合契约**：Mock 数据的结构必须与 `docs/` 下的接口文档完全一致
2. **Mock 必须有版本**：Mock 数据上标注基于哪个版本的契约（如 `// Mock v1.0 - based on api-contract.md Day3`）
3. **Mock 必须可切换**：代码中通过环境变量或配置开关切换 Mock / 真实实现
4. **Mock 及时淘汰**：一旦真实实现完成，立即替换 Mock，并在站会上同步

### 8.2 接口防扯皮规则

**规则 1：契约锁定后变更成本极高**
- GameState Schema 在 Day 2 下午 6 点锁定
- API 契约在 Day 3 下午 6 点锁定

**规则 2：Mock 先行**
- B 没做完时，A 用 Mock 数据并行开发

### 9.2 Agent 质量评估标准

人员C 需要定期评估 Agent 输出质量，使用以下评分表：

| 维度 | 权重 | 评分标准（1-5分） |
|---|---|---|
| **叙事风格** | 30% | 1=现代白话，5=古风修仙语感 |
| **角色一致性** | 25% | 1=NPC 说现代话/OOC，5=完全符合角色卡 |
| **状态一致性** | 20% | 1=出现与 GameState 矛盾的描述，5=完全基于当前状态 |
| **格式正确性** | 15% | 1=JSON 解析失败，5=完美符合格式 |
| **选项合理性** | 10% | 1=选项与场景无关，5=选项自然、有意义 |

**目标**：MVP 阶段平均分 >= 3.5，单项不低于 2 分。

## 10. 风险管理与应对策略

### 10.1 技术风险

| 风险 | 可能性 | 影响 | 应对策略 |
|---|---|---|---|
| **LLM API 不稳定/延迟高** | 高 | Agent 叙事卡顿 | 1. 超时降级（3秒无响应返回预设文案）<br>2. 本地模型备用（Ollama + qwen2.5）<br>3. 缓存常见场景输出 |
| **LLM 输出格式不稳定** | 高 | Parser 失败 | 1. Prompt 中强制 JSON 格式 + 示例<br>2. Parser 3 次重试 + Markdown 提取<br>3. 降级为预设文案 |
| **GameState Schema 频繁变更** | 中 | 全组返工 | 1. Day 2 强制锁定<br>2. 变更必须开 PR 全组 Review<br>3. Pydantic 自动校验 |
| **自由输入理解错误** | 中 | 玩家体验差 | 1. 意图分类 + 映射规则（非完全 LLM 决定）<br>2. 置信度低时返回通用回应<br>3. 收集玩家输入数据优化 Prompt |
| **前端性能差** | 中 | 体验差 | 1. requestAnimationFrame 打字机<br>2. 虚拟滚动处理长日志<br>3. 组件级 memo |
| **内容配置格式错误** | 中 | 引擎解析失败 | 1. `validate_content.py` 提交前自动运行<br>2. CI 中加入配置校验 |

### 10.2 进度风险

| 风险 | 可能性 | 影响 | 应对策略 |
|---|---|---|---|
| **Week 1 未打通端到端** | 中 | 全部延期 | 1. Day 5 铁 deadline，未达标周末加班<br>2. 提前预警机制（Day 3 评估风险）<br>3. 降级方案：固定文案 + 状态变化 |
| **某人进度严重落后** | 中 | 阻塞依赖方 | 1. 站会每日同步，4小时阻塞升级<br>2. 关键路径优先保障（人员B）<br>3. 必要时调整范围 |
| **内容产出不足** | 低 | 游戏空洞 | 1. 人员D Week 1 就开始产出<br>2. 预设文案兜底<br>3. MVP 只要求 5-8 个事件 |
| **Agent 叙事质量不达标** | 中 | 核心体验差 | 1. 持续优化 Prompt（加入更多示例）<br>2. 准备降级方案（固定文案 + 少量动态填充）<br>3. 换用更强的 LLM 模型 |

## 11. 开发规范

### 11.1 Git 工作流

```
main 分支（永远可运行）
  │
  ├── frontend/dev   (人员A)
  ├── backend/dev    (人员B)
  ├── agent/dev      (人员C)
```

- 每人从 `main` 切出自己的开发分支
- 功能完成后发 Pull Request，至少 1 人 Review
- **每天下班前 push 到远程**，即使功能未完成
- 里程碑日（Day 5, Day 12, Day 19）合并到 `main` 并打标签

### 11.2 代码规范

- **Python**：Black 格式化，类型注解，docstring
- **TypeScript**：ESLint + Prettier，严格模式
- **配置**：YAML 统一 2 空格缩进，字段名 snake_case
- **命名**：
  - 文件：snake_case（Python）/ camelCase（TS）
  - API 端点：kebab-case
  - 配置字段：snake_case

### 11.3 文档规范

- 所有接口契约放在 `docs/` 目录
- 接口变更必须同步更新文档，**文档与代码不一致以文档为准**
- README.md 包含：项目简介、启动步骤、技术栈、目录结构
