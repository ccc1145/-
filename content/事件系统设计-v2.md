# 事件系统设计 v2.0（世界书增强版）

## 一、核心理念

### 当前系统的问题

v1 的事件系统本质上是**分支选择树**——每个 scene 有预设 choices，choices 通过 `next_scene` 硬编码跳转。`free_input_enabled` 虽然存在，但玩家的自由输入并不会影响事件走向，因为所有路径都是预定义的。

这不是"AI 驱动的叙事"，而是"AI 渲染的 Choose Your Own Adventure"。

### 改造目标

充分利用 LLM 的叙事能力，让玩家 **真正自由地参与事件**，同时保证核心剧情框架不脱轨。核心策略：**大事件管骨架，小事件填血肉，自由探索是日常，世界书供弹药**。

---

## 二、双层架构

### 大事件（Major Events）

**定义**：游戏主线，时代洪流。由世界状态驱动的关键节点，个人难以改变趋势，但**个人选择决定自己的结果**。

**特征**：
- 触发条件：游戏阶段（flags）或世界状态改变
- 叙事方式：Agent 按预设场景框架渲染，但玩家有**真实的关键抉择**
- 玩家输入：预设选项（2-4个）+ 关键节点的自由输入
- 结果：各选项有明确的效果（属性/flag/NPC好感），但**结果描述由 Agent 自由生成**
- 不可跳过：大事件发生时，玩家必须参与

**示例**：入门试炼、拜师仪式、门派大比、宗门危机、天灾降临

### 小事件（Minor Events）

**定义**：玩家驱动的即兴互动。在自由探索中自然触发，**高度自由，AI 全权主导叙事**。

**特征**：
- 触发条件：玩家位置 + 属性 + NPC好感 + 随机因子
- 叙事方式：Agent 收到一个"事件骨架"（背景、冲突、NPC），由 Agent 自由展开
- 玩家输入：完全自由输入，Agent 理解意图并生成叙事
- 结果：Agent 根据玩家行为判断成功/失败，施加效果
- 玩家可以选择忽略（但可能错过奖励）

**示例**：后山发现神秘草药、师兄请你帮忙、与林玄的修炼切磋、遇到受伤的灵兽

### 日常事件（Daily Events）

**定义**：纯粹的自由探索状态。没有预设冲突，玩家完全自主行动。

**特征**：
- 触发条件：无大事件/小事件进行中时，默认就是日常状态
- 玩家可以做：移动、交谈、修炼、炼丹、探索、休息
- Agent 持续叙事，小事件潜力池随时可能触发

## 三、世界书系统（World Book）

### 什么是世界书？

借鉴 SillyTavern 的 Lorebook 概念，世界书是一个**动态知识注入系统**：

- 世界观知识被拆成数百个**知识碎片（Entry）**
- 每个碎片有**触发关键词（Keys）**
- 引擎扫描当前对话/场景，匹配到关键词后，将对应知识注入 Agent Prompt
- **只注入相关的知识**，不相关的知识不浪费 Token

### 为什么需要世界书？

**问题**：小事件和自由探索中，Agent 需要大量世界知识来生成合理的叙事。如果一次性把所有知识塞进 Prompt：
- Token 浪费严重
- Agent 被无关信息干扰
- 叙事质量下降

**解决**：世界书按需注入，Agent 始终获得"恰到好处"的知识支持。

### 世界书核心机制

```

├─────────────────────────────────────────────────────────────┤

│   └── 当前激活的 NPC 名称                                     │
│        │                                                     │
│        ▼                                                     │
│   ┌────────────────────────┐                                │
│   │ 关键词匹配              │                                │
│   │ 遍历所有知识碎片         │                                │
│   │ 检查 keys 是否命中      │                                │
│   │ 应用 selective logic   │                                │
│   └──────────┬─────────────┘                                │
│              │                                               │
```

### 知识碎片结构

```json
{
  "entry_id": "qingyun_history",
  "name": "青云门历史",
  "keys": ["青云门", "青云", "建派", "祖师"],
  "secondary_keys": ["历史", "由来"],
  "logic": "any_key",
  "content": "青云门建派三百年，出过三位元婴老祖...",
  "category": "world_lore",
  "weight": 200,
  "scan_depth": 4,
  "sticky": 0,
  "cooldown": 3,
  "probability": 1,
  "position": "after_scene",
  "max_tokens": 150
}
```

### 知识分类

| 分类 | 用途 | 注入位置 |
|------|------|---------|
| `world_lore` | 世界观、历史、地理 | after_scene |
| `npc_lore` | NPC 性格、背景、关系 | after_scene |
| `rule_knowledge` | 修炼规则、境界、功法 | after_scene |
| `event_effect` | 事件持续效果（sticky） | before_scene |
| `location_info` | 地点描述、环境细节 | after_scene |

### 与大事件/小事件的配合

**大事件**：
- 场景 `agent_guidance` 提供静态细节（setting_details、sensory_details）
- 世界书提供动态知识（如果玩家提到"炼丹"，注入炼丹知识）
- **静态 + 动态 = 完整的叙事素材**

**小事件**：
- `free_narrative_hints` 提供事件骨架
- 世界书根据玩家输入动态补充知识
- Agent 获得充足但不过量的信息，自由发挥叙事

**日常模式**：
- 玩家自由输入 → 扫描关键词 → 注入相关知识
- Agent 根据注入的知识生成连贯的叙事
- 小事件在自然对话中触发

### Prompt 构建流程

```
1. 基础系统指令（固定）
2. 角色卡信息（当前 NPC）
3. 当前场景描述（事件/自由探索）
4. 【世界书注入】← 动态知识区
5. 玩家输入
6. 对话历史
```

## 六、内容创作规范

### 新增知识碎片

1. 在 `content/world_books/` 下的 JSON 文件中追加条目
2. `keys` 是触发关键词，尽量覆盖玩家可能提到的同义词
3. `content` 要简洁但信息完整，控制在 `max_tokens` 范围内
4. `weight` 越高，优先级越高（重要知识设高权重）
5. `sticky` 用于事件持续效果（如：某事件后产生持续影响）
6. `cooldown` 防止同一知识反复触发
7. `probability < 1` 用于稀有知识（如：飞升真相只在特定条件下低概率触发）

### 知识碎片设计原则

- **原子性**：每个碎片只讲一件事，不要把太多信息塞进一个碎片
- **关键词覆盖**：keys 要覆盖玩家可能使用的各种说法
- **内容简洁**：Agent 不需要读论文，需要快速获取关键信息
- **相互独立**：碎片之间不应有强依赖，每个碎片可以独立注入

## 七、迁移计划

### 已有内容处理

| 内容 | 处理方式 |
|------|---------|
| 5 个大事件 | 归入 `events/major/`，添加 `type: "major"` |
| 3 个 NPC 角色卡 | 保留在 `npcs/`，knowledge 字段可逐步迁移到世界书 |
| 物品配置 | 保留在 `items/`，不变 |
| 世界观设定文档 | 作为世界书条目的素材来源 |

### 人员 B 接口变更

引擎需要新增世界书扫描模块：
1. 读取 `content/world_books/*.json` 加载所有知识碎片
2. 每次构建 Prompt 前，扫描当前上下文匹配关键词
3. 应用生命周期过滤（sticky/cooldown/probability）
4. 按权重排序，按 Token 预算截断
5. 将匹配到的知识注入到 Prompt 的指定位置
